Другие посты в этой серии были про то, как вывести пиксели на экран —
как подружить Servo с SDL2, как победить два GL-контекста и старый драйвер. В конце концов, приложение запускается и сайт. И вот ты сидишь, держишь его в
руках и думаешь, а как этим управлять.

Потому что веб-браузер опирается на два устройства ввода, которых у
консоли нет: мышь и клавиатуру. По ссылкам кликают мышью. URL набирают на
клавиатуре. Наведение, фокус, выделение текста, прокрутка — всё взаимодействие построено на курсоре, которым ты целишься, и сотне клавиш,
которые ты нажимаешь. У ретро-консоли есть крестовина (D-pad), несколько кнопок, и *может быть* один-два аналоговых стика. И всё. У некоторых даже стиков нет.

Так что вопрос, на который отвечает этот пост: как управлять браузером —
кликнуть по ссылке, набрать поисковый запрос, переключить вкладку, прокрутить
страницу — кнопками Game Boy? Ответ — это один и тот же приём, повторённый снова и
снова, пока весь интерфейс не будет пересобран от кнопок.

## Одна идея: намерения, а не кнопки

Ошибка, которой я избежал сразу же, — это зашить «A кликает, B возвращает назад».
Такое разваливается в тот же миг, когда встречаешь второе устройство, потому что
ни одна пара этих консолей не сходится даже в том, какие у них вообще кнопки. У
TrimUI Smart Pro два стика. У Anbernic RG35XX SP стиков *нет* совсем. Привяжи
интерфейс к физическим кнопкам — и ты написал интерфейс ровно для одного
устройства.

Поэтому слой ввода в retsurf вообще не говорит о кнопках. Он говорит о
**намерениях** — перечислении `Action` — а настраиваемая таблица сопоставляет
физические жесты с ними (`src/event/bindings.rs`):

```rust
pub enum Action {
    Confirm,
    Osk,
    // ...
    Scroll,
}
```

Вся задача слоя геймпада — превратить события контроллера SDL в одно из
этих намерений и передать его роутеру, который решает, что оно
*значит* в текущем контексте (`src/event/gamepad.rs`):

> Переводит сырой ввод контроллера в `AppCommand` — и ничего больше. Сопоставляет
> физические элементы управления с намерениями через настраиваемую таблицу
> `Bindings`; *что* делает каждое намерение — решает центральный роутер.

Именно эта прослойка делает всё компонуемым. «Кликнуть» — это
намерение; куда оно попадёт (ссылка на странице, кнопка на тулбаре, строка меню) —
решает роутер.

### Жест — это больше, чем нажатие

Десять-одиннадцать кнопок — недостаточно для целого браузера. Поэтому каждая
кнопка используется болеше чем для одной привязки. Жест — это
тап, удержание или аккорд, и значения по умолчанию опираются на все три
(`src/event/bindings.rs`):

```rust
("a", Action::Confirm),
("b", Action::Cancel),
// ...
("hold:start", Action::Reload),
```

Маппинг этих жестов — это машина состояний, и вся тонкость в таймингах.
Кнопка, у которой *единственная* привязка — это тап, может сработать сразу по
нажатию: ждать нечего. Но кнопка, которая ещё несёт `hold:` или участвует в
аккорде, в момент нажатия неоднозначна, поэтому её приходится отложить
(`src/event/gamepad.rs`):

> Кнопка, у которой единственная привязка — тап, срабатывает сразу по нажатию. Кнопка, которая ещё несёт привязку `hold:` или участвует в аккорде, в
> момент нажатия неоднозначна, поэтому она *откладывается*: тап срабатывает на
> отпускании (если оно случилось раньше `hold_ms`), действие удержания срабатывает
> по истечении порога, а аккорд срабатывает в тот момент, когда нажимается вторая
> кнопка.

То есть `hold_ms` — это развилка: отпустил раньше — значит
тапнул; перешагнул порог — значит удержал. Цена в том, что тап отложенной кнопки
теперь срабатывает на *отпускании*, а не на нажатии — пара миллисекунд задержки — что на практике незаметно и позволяет поставить второе действие на кнопку.

## Три способа кликнуть по ссылке

Клик — это задача, которую веб требует больше всего и которую консоль решает
хуже всего. У retsurf три решения.

### 1. Bиртуальный курсор

Самый буквальный перевод фразы «нет мыши» — это «подделай мышь». Левый стик
двигает точку по экрану; A кликает там, где точка. Позиция курсора — это просто
ограниченная рамкой точка, которую каждый кадр подталкивает отклонение стика, а
крестовина свёрнута в тот же вектор, так что работает как грубый запасной вариант
(`src/event/gamepad.rs`):

```rust
fn aim(&self) -> (f32, f32) {
    (
        (self.left.0 + self.dpad.0).clamp(-1.0, 1.0),
        (self.left.1 + self.dpad.1).clamp(-1.0, 1.0),
    )
}
```

А вот сам клик — это место, где всё встречается с Servo. Никакого системного
указателя двигать нельзя — поэтому нажатие *синтезирует* события мыши, которых
ждёт движок, но только когда курсор реально над страницей, а не над тулбаром
(`src/app/router.rs`):

```rust
fn primary_action(&mut self, pressed: bool) {
    if self.ui.cursor_over_browser() {
        let (x, y) = self.ui.cursor_browser_rel();
        self.browser
            .handle_input(servo::InputEvent::MouseMove(into_mouse_move_event(x, y)));
        let event = into_mouse_button_event(sdl2::mouse::MouseButton::Left, x, y, pressed);
        self.browser
            .handle_input(servo::InputEvent::MouseButton(event));
    } else {
        self.ui.click_ui(pressed, &self.window);
    }
}
```

`MouseMove`, чтобы выставить состояние наведения, затем `MouseButton` down/up —
ровно та последовательность, что выдала бы настоящая мышь, изготовленная из стика
и кнопки. Это работает, и это честно, и это *медленно*: целиться стиком по
результату поиска — худшая часть всего UX. Хуже того, на устройстве без стика это
мёртвый груз. Виртуальный курсор — это пол, а не потолок.

### 2. Режим прокрутки

Это `Action::Scroll` на тапе Start существует ради устройств без стиков. Без
правого стика страницу было бы невозможно прокрутить.
Поэтому тап по Start *переключает* режим, в котором левый стик и крестовина
переосмысляются как прокрутка страницы, а не движение курсора. Это единственное действие, которое слой геймпада обрабатывает сам, а не
маршрутизирует, потому что оно меняет смысл каждого последующего кадра. Один
физический ввод, два режима, переключаемые кнопкой, — повторяющееся решение на
настолько ограниченном железе.

### 3. Подсказки-ссылки с бейджами

Настоящий ответ позаимствован у [Vimium](https://vimium.github.io/): не целься в
ссылки, *подпиши* их. Входишь в режим подсказок — и каждый кликабельный элемент на
странице получает маленький бейдж; выбираешь бейдж, и он кликает. Без прицеливания.

Сначала надо найти кликабельные элементы. retsurf инжектит JavaScript в активную
страницу через Servo и считывает обратно плоский список прямоугольников
(`src/browser/mod.rs`):

```javascript
const els = document.querySelectorAll(
    'a[href], button, input:not([type="hidden"]), select, textarea, summary, ' +
    '[onclick], [role="button"], [role="link"], [role="tab"], [contenteditable="true"]'
);
```

Каждый видимый элемент возвращается как его прямоугольник во
вьюпорте плюс URL, с потолком в 150, чтобы страница из множества ссылок не перегрузила
IPC-канал. Пока что это Vimium. А вот где консоль его адаптирует.

Vimium подписывает свои подсказки *буквами* — набираешь `fj`, чтобы кликнуть. У
retsurf клавиатуры нет. Так что алфавит — это не буквы, это сами кнопки. Их восемь,
поскольку A и B уже заняты (подтверждение и отмена):

```rust
pub enum Sym {
    X, Y, L1, R1, Up, Down, Left, Right,
}
```

Восемь символов означают, что коды подсказок — это просто восьмеричные числа, где
каждый «разряд» нарисован как кнопка, которую надо нажать. Коды остаются настолько
короткими, насколько позволяет страница, — одна кнопка на 8 ссылок, две на 64, три
на 512, — и *ближайшие* подсказки получают самые короткие коды, упорядоченные по
расстоянию от верха вьюпорта (`src/overlay/hints.rs`):

```rust
fn assign_codes(hints: &[Hint], anchor: (f32, f32)) -> Vec<Vec<Sym>> {
    let n = hints.len();
    let len = if n <= 8 { 1 } else if n <= 64 { 2 } else { 3 };
    let mut order: Vec<usize> = (0..n).collect();
    order.sort_by(|&a, &b| {
        dist2(hints[a].center(), anchor).total_cmp(&dist2(hints[b].center(), anchor))
    });
    let mut codes = vec![Vec::new(); n];
    for (rank, &idx) in order.iter().enumerate() {
        codes[idx] = nth_code(rank, len);
    }
    codes
}
```

Дальше ты нажимаешь кнопки, и набранный буфер сужает поле. Сопоставление
получается приятнее, чем звучит, потому что коды все одной длины и уникальны: так
что полный буфер, который всё ещё *чему-то* соответствует, соответствует ровно
одной вещи — никакого Enter для разрешения неоднозначности не нужно
(`src/overlay/hints.rs`):

```rust
pub fn push_sym(&mut self, s: Sym) -> HintInput {
    self.typed.push(s);
    if !(0..self.codes.len()).any(|i| self.has_prefix(i, &self.typed)) {
        self.typed.clear();
        return HintInput::NoMatch;
    }
    let len = self.codes.first().map_or(0, Vec::len);
    if self.typed.len() >= len {
        let idx = (0..self.codes.len())
            .find(|&i| self.codes[i] == self.typed)
            .expect("a full-length prefix match is an exact match");
        self.typed.clear();
        return HintInput::Activate(idx);
    }
    HintInput::Pending
}
```

На экране, пока ты нажимаешь кнопки, каждый бейдж, чей код больше не подходит под
твой ввод, гаснет до 16% непрозрачности. Это читается
так, будто страница помогает тебе целиться, в чём и весь смысл.

И поскольку процесс двухэтапный — грубый прыжок крестовиной, чтобы *передвинуть
выделение*, затем нажатие, чтобы *подтвердить*, — режим подсказок тоже должен
прокручивать. Если ты сдвигаешь выделение за нижний край, а дальше идти некуда,
страница должна прокрутиться и показать больше подсказок, а не застрять. Это
большая часть `hints_nav`, и в ней прячется самый раздражающий баг этой фичи
(`src/app/router.rs`):

```rust
let (sx, _) = self.ui.hints_selected_center()
    .unwrap_or_else(|| self.ui.cursor_browser_rel());
let sy = height / 2.0;
let chunk = dy as f32 * height * HINT_EDGE_SCROLL_FRACTION;
self.browser.scroll(0.0, chunk, sx, sy);
```

Баг: Servo решает, *какой* элемент прокручивать, делая hit-test по точке, из
которой ты прокручиваешь. Подсказка, сидящая у самого верхнего края, обычно
находится внутри липкого заголовка (sticky header) — строки поиска DuckDuckGo,
навигации сайта, — а липкий заголовок не прокручивает документ. Так что прокрутка
«из» этой подсказки не двигала ничего, и страница выглядела замёрзшей. Исправление
— сохранить горизонтальную колонку подсказки (чтобы подсказка, живущая во
вложенном скроллере, всё же прокручивала *его*), но делать hit-test по середине
вьюпорта по вертикали, что попадает на основной контент. После того как прокрутка
устаканится, подсказки пересобираются, и выделение заново привязывается к только что
открывшемуся краю. Настоящий баг конкретно для консоли — он существует только потому, что курсор
не настоящий указатель, а прокрутку приходится наводить.

## Печать без клавиатуры

Клик решён, второе недостающее устройство — клавиатура. Без печати браузером не
попользуешься: хотя бы раз надо набрать URL или запрос, — поэтому retsurf включает в себя экранную клавиатуру: сетку egui, по которой ты ходишь
крестовиной. Состояние ровно такое, как ты и подумал: строка, столбец и стопка
раскладок (`src/overlay/osk.rs`):

```rust
pub struct Osk {
    visible: bool,
    caps: bool,
    caret: usize,
    row: usize,
    col: usize,
    layouts: Vec<Layout>,
    lang: usize,
}
```

Навигация крестовиной — это просто ограниченное рамкой движение по сетке:

```rust
fn move_sel(&mut self, dx: i32, dy: i32) {
    let rows = self.layout().keys.len() as i32;
    self.row = (self.row as i32 + dy).clamp(0, rows - 1) as usize;
    let cols = self.layout().keys[self.row].len() as i32;
    self.col = (self.col as i32 + dx).clamp(0, cols - 1) as usize;
}
```

Но добираться до каждой клавиши может быть долго, поэтому кнопки, которые чаще всего используются, получают отдельные шорткаты в обход сетки: X — это Backspace,
Y — Space, L2 — моментальный Shift, R2 — Enter. Раскладок тоже больше одной — QWERTY и кириллица ЙЦУКЕН, — потому что «набрать URL» и «набрать поисковый запрос по-русски» — разные
задачи.

Деталь, которая делает экранную клавиатуру переиспользуемой, — то, что она не
знает, куда идёт текст. Она пишет в таргет, которую передаёт вызывающая сторона
(`src/overlay/osk.rs`):

```rust
pub enum OskTarget<'a> {
    AddressBar,
    Prompt(&'a mut String),
    // ..
    Page,
}
```

Большинство таргетов — это просто буфер, который клавиатура правит на месте.
Интересная — `Page`: печать в сфокусированный `<input>` на сайте, где никакого
буфера для правки нет, — так что каждый символ превращается обратно в
синтезированную пару key-down/key-up и отдаётся Servo, зеркальное отражение того,
как курсор синтезирует события мыши.

## Меню, по которым ходишь крестовиной

Вкладки, закладки, история, загрузки — обвязка браузера — это полноэкранные
оверлеи, а не выпадашки, потому что в выпадашку нужно целиться. Навигация — та же
модель выделения, что и везде: крестовина двигает внутри активной секции, шифты
переключают секции, а выбранная строка сама прокручивается в зону видимости, чтобы
выделение за пределами экрана никогда не терялось (`src/overlay/menu.rs`):

```rust
pub fn move_sel(&mut self, dy: i32) {
    match self.section {
        Section::Bookmarks => self.bookmarks.move_sel(dy),
        // ..
        Section::Tabs => {
            let last = self.tab_count as i32;
            self.tab_selected = (self.tab_selected as i32 + dy).clamp(0, last) as usize;
        }
    }
}
```

Ничего хитрого здесь нет, и в этом весь смысл: как только «выделение плюс
подтверждение» становится идиомой, каждый элемент в приложении — это набор строк. Курсор,
подсказки, клавиатура, меню — все они суть выделение-и-подтверждение в разных
местах.

## Настройки

Фичу, которую я откладывал дольше всего - возможность переназчать кнопки через настройки. Значения по умолчанию — это догадка, но
догадка не может быть правильной для каждого устройства и пользователя — так что привязки должны
быть редактируемыми.

Вот только: конфиги правят клавиатурой, а вся идея этого поста в том, что
клавиатуры *нет*. `bindings.toml`, который можно отредактировать, только вынув
SD-карту и примонтировав её на компьютере, — это не та фича, которую
пользователь консоли хотел бы. Так что изменение должно происходить *в
приложении*, а это значит максимально чистый UX: чтобы
привязать действие, ты нажимаешь ту кнопку, которую хочешь.

Этот процесс захвата — небольшое переключение режима. Когда ты активируешь
«добавить привязку», оверлей настроек запоминает, какое действие слушает
(`src/overlay/settings.rs`):

```rust
pub struct Settings {
    // ...
    capturing: Option<Action>,   // Some(action) while waiting for a gesture
}
```

Пока это `Some`, обработчик событий перехватывает ввод *до* того, как он
дойдёт до обычной диспетчеризации, — иначе нажатие A для его привязки просто
запустило бы уже существующее у A действие «подтвердить». Для клавиатурной стороны
(захват работает и для клавиатурных шорткатов — на устройствах, где клавиатура
есть) он ловит нажатие, игнорирует модификаторы и превращает остальное в
строку-жест (`src/event/handler.rs`):

```rust
if ui.settings_capturing() {
    match event {
        Event::KeyDown { keycode: Some(kc), keymod, repeat: false, .. } => {
            if kc == Keycode::Escape {
                commands.push(/* cancel capture */);
            } else if is_modifier_key(kc) {
                return;                          // wait for the real key
            } else if let Some(gesture) = bindings::format_key(kc, keymod) {
                commands.push(/* CaptureBinding { gesture, keyboard: true } */);
            }
            return;
        }
        // ...swallow key-up and text-input so nothing leaks through
    }
}
```

Сторона геймпада проверяет *то же самое* разрешение тап/удержание/аккорд, что и
живой UI, так что ты можешь захватить «удержать Select» или «L1+R1» ровно так, как
ты потом будешь это выполнять, и форматирует результат тем же словарём, которым
записаны значения по умолчанию (`src/event/bindings.rs`):

```rust
pub(crate) fn button_name(button: Button) -> Option<&'static str> {
    Some(match button {
        Button::A => "a",
        Button::X => "x",
        // ...
        _ => return None,
    })
}
```

Есть и последняя деталь — режиму захвата нужен аварийный выход — а что, если ты открыл его случайно, и нет
клавиши Esc, чтобы отменить? На устройстве, где нет ничего, кроме геймпада, ответ —
таймаут:

```rust
/// Auto-cancel binding capture if nothing is pressed for this long — the handheld
/// escape hatch (there's no Esc), so an accidental "add" doesn't trap input.
const CAPTURE_TIMEOUT: Duration = Duration::from_secs(6);
```

Отсутствие клавиатуры — это не проблема, которую решаешь один раз в строке
URL; это ограничение, которое следует за тобой в каждый угол интерфейса, включая то
самое меню, где ты настраиваешь кнопки.

## Бонус: тот же приём на клавиатуре

Всё до этого предполагало, что клавиатуры нет, — и для консоли это правда.
Но retsurf ещё работает на Android — и на
телефоне клавиатура внезапно снова в игре: Bluetooth-клавиатура или аппаратный
QWERTY телефона. Вот где окупается трюк «намерения, а не
кнопки». Клавиатура — это не вторая схема управления, которую надо проектировать и
поддерживать; это вторая таблица привязок поверх тех же самых `Action`.

Это буквально структура `bindings.toml` — две таблицы, один словарь
(`src/event/bindings.rs`):

```rust
pub struct Store {
    pub gamepad: BTreeMap<String, String>,
    pub keyboard: BTreeMap<String, String>,
}
```

А `src/event/keyboard.rs` сам описывает себя как зеркало слоя геймпада:

> Переводит ввод клавиатуры в `AppCommand` — клавиатурный двойник
> `gamepad`. Владеет таблицей `[keyboard]` в `bindings.toml` и применяет правила
> срабатывания: шаги `nav_*` требуют открытого оверлея (и авто-повтор при
> удержании), обычные шорткаты (без Ctrl/Alt) глушатся, пока фокус в чём-то
> редактируемом, а меню / оверлей подсказок первыми получают свои фиксированные
> клавиши. Всё, что не поглощено, уходит на страницу как событие клавиатуры Servo.

Дефолтная раскладка — то, чего пальцы привыкшего к браузеру человека и так ждут, с
уклоном в Vimium (`src/event/bindings.rs`):

```rust
("ctrl+r", Action::Reload),
("ctrl+b", Action::Bookmark),
// ..
("f", Action::Hints),
("l", Action::NavRight),
```

`f` привязана прямо к `Action::Hints` — тому же намерению, что и кнопка Y. Жмёшь
`f` — включается режим подсказок; дальше двигаешь выделение через `hjkl` или
стрелки и жмёшь Enter — там, где геймпад набрал бы кнопочную комбинацию из раздела
выше. Два способа выбрать подсказку, один общий `activate_hint`.

Cложная часть браузера на клавиатуре — конфликт, которого у геймпада
нет: обычная `f` должна *открывать подсказки*, пока ты читаешь, но печатать букву
*f*, когда ты в поле поиска. Голая буква не может быть одновременно шорткатом и
символом, пока что-то не решит, чем именно, — на каждое нажатие. Это решение
умещается в четыре строки (`src/event/keyboard.rs`):

```rust
fn lookup(&self, key: &KeyEvent, overlay: bool, typing: bool) -> Option<Action> {
    let (action, plain) = self.bindings.lookup(key.kc, key.keymod)?;
    let fire = if action.is_nav() {
        overlay
    } else {
        !key.repeat && (!plain || !typing)
    };
    fire.then_some(action)
}
```

Привязка без модификатора (`plain`) срабатывает, только когда фокус не в
редактируемом поле (`!typing`); как только текстовое поле или адресная строка
получают фокус, `f` и `hjkl` перестают быть шорткатами и проходят как символы. Шаг
`nav_*` срабатывает только при открытом оверлее, так что стрелки управляют меню, но
прокручивают страницу, когда меню нет. Ctrl-комбо срабатывают безусловно — `Ctrl+R`
перезагружает, печатаешь ты или нет. Три правила — и клавиатура перестаёт воевать с
текстовым полем.

Android добавляет в ту же схему ещё один специфичный для устройства элемент.
Жестовый или аппаратный Back телефона приходит как событие клавиши и маппится на
фокусо-зависимое намерение `Cancel` — закрыть открытый оверлей или вернуться назад
на странице (`src/event/keyboard.rs`):

```rust
if key.kc == Keycode::AcBack {
    if key.pressed && !key.repeat {
        commands.push(AppCommand::Input(InputCommand::Cancel));
    }
    return;
}
```

Так что свайп-Back на Android, тап B на геймпаде и Backspace на клавиатуре — три
не связанных физических события, которые все сводятся к одному намерению —
`Cancel`, — и роутер делает для каждого одно и то же, с учётом фокуса. Всё, что не
забрала ни одна таблица, проваливается на страницу настоящим событием клавиатуры
Servo, так что ввод в веб-форму просто работает. А клавиатурная раскладка
перепривязывается тем же процессом захвата из прошлого раздела — его половиной
`[keyboard]`, — так что пользователь QWERTY-телефона перекраивает свои шорткаты,
тоже не трогая конфиг.

Изначально я просто хотел добавить управление с геймпада. В результате появился слой ввода, который одинаково работает с геймпадом, жестами на тачскрине и полноценной клавиатурой. Ни один из этих способов ввода никогда не был привязан к интерфейсу напрямую. Все они были привязаны к намерениям пользователя.
