[retsurf](https://github.com/mxmgorin/retsurf) — объединяет 3 больших компонента: **Servo** отрисовывает веб, **SDL2** владеет окном и вводом на «голом»
железе, а **egui** рисует поверх собственный интерфейс браузера. В каждом из них нет ничего сложного. Сложность в том, чтобы заставить все три делить *один GPU* на
устройстве без композитора и со старым графическим драйвером.

Это полный разбор: как соединяются Servo и SDL2 и
все причины, по которым это ломалось по пути к единственному общему
GL-контексту. По порядку — потому что порядок и есть сюжет: каждая проблема — это то,
что толкало архитектуру к следующей итерации.

## Первая попытка — surfman из хендла окна SDL2

Самое очевидное, что приходит на ум первым — использовать то, что Servo релизовывает сам. Servo не просит у тебя пиксели — он просит `RenderingContext`, и его
библиотека поверхностей [surfman](https://github.com/servo/surfman) с радостью
построит *свой* собственный GL-контекст из «сырого» хендла окна — она хочет сама
управлять соединением с GPU. А крейт `sdl2` умеет отдать этот хендл через
`raw-window-handle`, так что связываешь одно с другим — и Servo сам владеет
соединением с GPU от начала и до конца.

На моём Linux-десктопе это сразу заработало — `sdl2` возвращает X11- (или Wayland-)
хендл, surfman строит из него контекст, и страницы отрисовывались. Выглядело как
решённая задача.

Потом я запустил тот же бинарник на консоли — и он умер сразу на старте. Эти
устройства (Knulli, muOS, ROCKNIX) работают на голом KMS/DRM без X11- или
Wayland-композитора, а Rust-крейт `sdl2`, по состоянию на 0.38, отдаёт
raw-window-handle для Wayland, X11, Win32 — но **не для DRM/GBM**. Так что surfman
было попросту не из чего строить контекст: вызов, который на десктопе возвращал
хендл, на устройстве возвращал пустоту. Дефолтный путь здесь не деградирует мягко —
его просто нет.

Это и задаёт всю архитектуру:

> **SDL2 обязан владеть единственным GL-контекстом** — он уже умеет создавать его
> на голом KMS/DRM через EGL/GBM, ровно так же, как любой другой SDL2-порт под
> консоли, — а **Servo рисует *в него*.**

## Вторая попытка — программная отрисовка

Поэтому SDL2 создаёт контекст сам. Вот он берёт владение на себя, в
`src/platform/window.rs`: он запрашивает **GLES**-контекст (Mali-блобы на этих SoC
говорят только на GLES; WebRender требует ≥ 3.0):

```rust
let gl_attr = video_subsystem.gl_attr();
if config.use_gles {
    // Mali blobs on RK3326/RK3566 expose GLES 3.2; WebRender needs >= 3.0.
    gl_attr.set_context_profile(GLProfile::GLES);
    gl_attr.set_context_version(3, 0);
}
gl_attr.set_double_buffer(true);

let window = video_subsystem.window("retsurf", w, h).opengl().resizable().build()?;
let gl_context = window.gl_create_context()?;
window.gl_make_current(&gl_context)?;
```

Затем загрузчик процедур SDL кормит *оба* GL-стека — `glow` для egui и `gleam` для
Servo/WebRender — из одного и того же контекста:

```rust
let glow_ctx = Arc::new(unsafe {
    glow::Context::from_loader_function(|name| {
        video_subsystem.gl_get_proc_address(name) as *const _
    })
});
let gl: Rc<dyn Gl> = unsafe {
    gleam::gl::GlesFns::load_with(|name| {
        video_subsystem.gl_get_proc_address(name) as *const _
    })
};
```

Две GL-привязки, один контекст. Запомни эту мысль — она источник половины проблем
ниже.

Когда окно и контекст готовы, я хотел доказать
работоспособность остального конвейера от начала и до конца, прежде чем снова бороться
с GPU. Поэтому использовал `SoftwareRenderingContext` из Servo
(офскрин-растеризатор на llvmpipe). Каждый кадр: Servo растеризует страницу на CPU,
retsurf вызывает `read_to_image()`, чтобы вытащить пиксели, загружает их как
текстуру egui, и egui компонует эту текстуру с панелью инструментов.

Это работало. Страницы появлялись. Но это неправильно по двум причинам:
требуется наличие llvmpipe на устройстве (CPU-растеризатор — ровно то, чего ты *не*
хочешь на слабом ARM-чипе), и на каждом кадре делается полный readback кадра плюс
повторная загрузка. Это был шаг, а не пункт назначения.

И, что критично: в одном потоке жили **два GL-контекста** — SDL-овский и
тот, который surfman поднимал для программного рендерера. Вот откуда начались
проблемы.

## Проблема №1 — конфликт eglBindAPI (паника при старте)

SDL создал **GLES**-контекст. Программный контекст surfman был на **десктопном
GL**. EGL отслеживает «текущий клиентский API» для каждого потока, и когда один
стек привязывает ES, а другой — десктопный GL, всё падало на старте.

Лечилось это переменной окружения, которая выставлялась до любой инициализации GL:
она велит surfman быть на GLES, чтобы оба стека сошлись (`src/lib.rs`):

```rust
if app_config.display.use_gles {
    // SDL creates a GLES context (sets the thread's EGL API to ES). Servo's
    // surfman context must use the same API or context creation fails, so
    // force surfman to GLES too. Must be set before any surfman/SDL GL init.
    std::env::set_var("SURFMAN_FORCE_GLES", "1");
}
```

То, что один стек расходится с другим во мнении, *какой GL* сейчас в игре,
оказалось повторяющейся темой — см. проблему №4.

## Проблема №2 — кэш make-current и неопределённый framebuffer

Это один из самых противных багов: *тысячи* ошибок и чёрное окно без всякой
видимой причины.

SDL кэширует, какой GL-контекст он *считает* текущим, чтобы пропускать лишний
`eglMakeCurrent`. Вот только surfman, занимаясь своей отрисовкой, вызывал
`eglMakeCurrent` напрямую и менял текущий контекст потока **за спиной SDL**. Кэш
SDL становился невалидным. Поэтому, когда egui собирался рисовать и просил SDL сделать
контекст окна текущим, SDL смотрел в свой кэш, решал «уже текущий» и *пропускал
настоящий `eglMakeCurrent`*.

Дальше egui рисовал в то, что было реально привязано, — в контекст surfman, а не в
окно, — и GPU сообщал `GL_FRAMEBUFFER_UNDEFINED`, а следом — поток
`GL_INVALID_FRAMEBUFFER_OPERATION`. И так каждый кадр.

Лечилось тем, чтобы инвалидировать кэш SDL перед повторной привязкой — так SDL
вынужден выдать настоящий `eglMakeCurrent`:

```
SDL_GL_MakeCurrent(window, NULL)   // clear SDL's "current context" cache
window.gl_make_current(&gl_context) // this actually calls eglMakeCurrent
```

Урок здесь: **когда две библиотеки меняют одно и то же скрытое
глобальное состояние, любой кэш, который одна из них держит —
мина.** SDL кэшировал текущий EGL-контекст; surfman переписывал его, не сказав SDL.

## Проблема №3 — паника при завершении

Когда отрисовка заработала, падал выход. `SoftwareRenderingContext` из Servo не
уничтожает свой surfman-контекст при drop, а у surfman есть проверка, которая
*паникует*, если контекст удалили без явного уничтожения:
`Contexts must be destroyed explicitly`. Так что чистый выход разворачивался в
панику.

Лечение, которое до сих пор в коде (`src/app/mod.rs`), — вообще не
запускать этот деструктор: корректно завершаем Servo там, где это важно (cookies,
localStorage), а остальное отдаём ОС:

```rust
self.browser.shutdown();   // flushes cookies / localStorage to disk

// Servo's rendering context doesn't destroy its surfman context on drop, which
// trips surfman's "destroy explicitly" guard and panics during unwinding. Exit
// before running destructors; the OS reclaims everything.
std::process::exit(0);
```

Неизящно, но правильно: процесс всё равно завершается, ядро освобождает ресурсы GPU, а
пропуск деструктора ничего не стоит.

## Проблема №4 — SDL и surfman на разных дисплейных серверах

Этот касается только на Wayland-*десктопе*, но он поучителен, потому что болезнь та
же, что у проблемы №1: два стека расходятся во мнении об окружении.

surfman выбирает свой дисплейный бэкенд по переменным окружения — если выставлен
`WAYLAND_DISPLAY`, он идёт в Wayland. SDL делает *свой* независимый выбор, и на
многих Wayland-десктопах SDL по-прежнему по умолчанию берёт **x11** (через
XWayland). Так surfman оказывается на Wayland, SDL — на x11, их GL-контексты не
могут сосуществовать, создание контекста у surfman падает → паника при старте.

Лечение заставляет SDL согласиться с surfman, когда оба на Wayland-десктопе, и не
лезет никуда в остальных случаях (`src/lib.rs`):

```rust
// surfman picks Wayland when WAYLAND_DISPLAY is set; SDL often defaults to x11.
// Align them. On the handheld (no WAYLAND_DISPLAY) this is skipped → SDL uses
// kmsdrm as intended. An explicit SDL_VIDEODRIVER always wins.
if std::env::var_os("SDL_VIDEODRIVER").is_none()
    && std::env::var_os("WAYLAND_DISPLAY").is_some()
{
    std::env::set_var("SDL_VIDEODRIVER", "wayland");
}
```

## Переосмысление — перестать иметь два контекста

У трёх из этих четырёх проблем (№1, №2, №3) одна общая первопричина: **в потоке было
два GL-контекста, дравшихся за глобальное состояние EGL.** Изначально я планировал
заставить surfman использовать контекст SDL, чтобы они стали одним. Но более чистым
решением оказалось противоположное — убрать surfman из пути отрисовки целиком.

Ключевое осознание: `servo::RenderingContext` — это *трейт*. Я не обязан
использовать реализацию surfman; я могу написать свою поверх единственного контекста
SDL. А WebRender рисует в **тот фреймбуфер, который привязан** после того, как он
вызвал `prepare_for_rendering()`. Так что я просто привязываю там свой собственный
объект (FBO), даю Servo рисовать в него, а egui отдаю цветовую текстуру
этого FBO, чтобы скомпоновать.

Это `SdlRenderingContext` в `src/platform/render.rs`. Вся реализация трейта
подозрительно маленькая:

```rust
impl RenderingContext for SdlRenderingContext {
    fn prepare_for_rendering(&self) {
        // WebRender renders into whatever FBO is bound here — so bind ours.
        self.gl.bind_framebuffer(gl::FRAMEBUFFER, self.fbo.get());
    }

    fn present(&self) {
        // No swap: egui composites the color texture into the window framebuffer.
    }

    fn make_current(&self) -> Result<(), surfman::Error> {
        // Single shared SDL2 GL context; already current on this thread.
        Ok(())
    }

    fn connection(&self) -> Option<surfman::Connection> {
        self.connection.clone()
    }
    // ...gleam/glow accessors, resize, read_to_image
}
```

Посмотри, что эти функции *не* делают. `make_current` — пустышка, возвращающая
`Ok(())`: контекст один и он уже текущий, так что не на что переключаться и не из-за
чего драться. `present` пуст — реальным свопом буфера владеет egui. С единственным
контекстом проблем №1, №2 и №3 просто не может случиться. (Теперь они на всякий
случай: `SURFMAN_FORCE_GLES` из №1 и выравнивание драйверов из №4 всё ещё
применяются только потому, что `connection()` *по-прежнему* зовёт в surfman — а это
завязка для финальной проблемы.)

## Как Servo и egui на самом деле соединяются, кадр за кадром

С новым способом — главный цикл становится:

```rust
// Render Servo into its FBO; egui composites that FBO's texture.
self.browser.paint();   // Servo/WebRender → our FBO (bound in prepare_for_rendering)
self.ui.update(...);    // egui builds the toolbar/overlays
// ...
self.draw();            // egui binds framebuffer 0, draws the FBO color texture, swaps
```

Один контекст, один FBO, одна передача текстуры, один своп. Это и есть весь мост
между Servo и SDL2 — и именно он наконец работает на голом железе консоли.

## Проблема №5 — работало на десктопе, но паниковало на реальном устройстве

Всё, что выше, было проверено на десктопе при GLES 3.2: ноль GL-ошибок, страницы
правильной стороной вверх. Я собрал под aarch64, скопировал на Mali-консоль — и оно
спаниковало на старте:

```
surfman .../egl_bindings.rs: egl function was not loaded
```

Виновник — тот самый вызов `connection()`. surfman 0.12 строит своё
соединение через `eglGetPlatformDisplay` — символ из **EGL 1.5**, — загружаемый
через `dlsym` на каждом Linux-бэкенде. Mesa на моём десктопе — EGL 1.5; Mali-блоб
устройства — **EGL 1.4** и этого символа попросту не имеет. Дальше surfman
*паникует* вместо того, чтобы вернуть ошибку, а регистрация пейнтера в Servo делает
`.expect()` на соединении — два слоя, оба выбравшие «упасть» вместо «справиться».

Соединение используется только для внешних изображений WebGL/WebGPU, без которых
консоль прекрасно живёт, поэтому лечение — позволить ему отсутствовать. `connection()`
в retsurf прощупывает surfman за `catch_unwind` (панику нельзя поймать через `match`)
и возвращает `None` при сбое; небольшой вендоренный патч Servo заставляет регистрацию
трактовать отсутствующее соединение как «WebGL выключен», а не паниковать. В реальной
релизной сборке весь путь WebGL спрятан за Cargo-фичей `webgl`, которая вырезается из
компиляции. Именно этот баг научил меня большему
всего, и его урок — вся причина, почему он стоил выходных: *драйверы твоего десктопа
— не целевая платформа, а библиотека, которая паникует на отсутствующей
необязательной возможности, лишает тебя плавной деградации.*

## Почётные упоминания (мелкие проблемы)

Не каждая проблема приводила к полному падению. Несколько таких, что съели по
полдня каждая:

- **Нельзя отладить то, чего не видишь.** Лаунчер консоли выбрасывает stderr, так
  что от паники не остаётся ничего, кроме кода выхода 101. retsurf ставит
  panic-хук, который дублирует сообщение *и бэктрейс* в файл, а ещё умеет писать
  логи в файл (`RETSURF_PANIC_FILE` / `RETSURF_LOG_FILE`). Первый баг на устройстве
  я смог диагностировать только *благодаря* этому — иначе проблема №5 была бы невидимым
  мгновенным выходом.
- **Оверлеи egui невидимы на первом кадре.** Свежесозданная egui-`Area`
  раскладывается невидимо на первом проходе (она измеряет саму себя), так что
  оверлей, открытый в ответ на ввод, не появился бы до *следующей* перерисовки — а
  цикл блокируется на вводе, так что её часто и не было. Лечится вызовом
  `request_repaint()` после создания оверлея, который форсирует второй кадр, реально
  его рисующий.
- **Чёрная полоса на самом первом кадре**, потому что размер вьюпорта устанавливается
  только после одного ресайза. Косметика, и всё ещё открытый пункт в списке уборки,
  а не решённый.

## Чему меня это научило

Четыре урока, примерно по убыванию боли:

1. **Крупные архитектурные решения за тебя принимает платформа.** «SDL2 владеет
   контекстом, Servo рисует в FBO» — я так не выбирал; так вышло из-за того,
   что у крейта sdl2 нет window-handle для DRM/GBM. Ограничение и оказалось
   архитектурой.

2. **Две библиотеки на одном скрытом глобальном состоянии — фабрика проблем.** Три
   проблемы из пяти — это борьба SDL и surfman за текущий EGL-контекст потока или
   за дисплейный бэкенд. Помогло не примирение, а *удаление одной из них* с горячего
   пути. А что самый простой вариант — один контекст и рендеринг в FBO — и есть
   лучший, стало ясно только после того, как схема с двумя контекстами как следует
   меня помучила.

3. **Десктоп усыпляет бдительность.** Mesa — свежая и всё прощает; драйвер на
   устройстве — замороженная минимальная заглушка EGL 1.4. Любая проблема, которая
   реально мешала релизу (как №5), на десктопе была не видна, а на устройстве
   вылезала мгновенно. Настоящим тестом оказалось только железо.

4. **Паниковать там, где ошибку можно обработать, — дорогой вид багов.** Отсутствие
   символа EGL 1.5 на GPU 2021 года — не редкость, но surfman вместо `Err`
   паниковал, и тремя слоями выше это превращалось в неустранимый сбой. Будь
   библиотекой, которая возвращает `Option`.

Развязка вышла будничной — и по ней понимаешь, что всё готово: устройство грузится,
и настоящая, напичканная JavaScript страница рисуется на Mali-GPU через один общий
GLES-контекст — без readback-ов, без GL-ошибок, без паник. Просто браузер на штуке,
которую делали под ROM-ы Game Boy.
