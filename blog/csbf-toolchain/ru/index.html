Язык без синтаксиса: компилятор Brainfuck на C#

2026-07-06 — C# · Compilers · Brainfuck · WASM — 8 мин чтения

После эмулятора Game Boy на Rust (oxGBC) и интерпретатора CHIP-8 на Go (ch8go) мне стало интересно попробовать другую сторону процесса. Оба этих проекта исполняют уже существующий байткод и следующим логичным шагом было написать программу, которая этот байткод сама создаёт. При этом хотелось выбрать максимально простой язык, чтобы основное внимание уделить именно компилятору, а не парсеру и синтаксису.

Brainfuck подходит для этого почти идеально. В языке всего восемь инструкций — + - < > . , [ ], байтовая лента и указатель. Никаких переменных, типов, функций или сложной грамматики. Благодаря этому почти вся сложность смещается туда, где действительно начинается работа компилятора: промежуточное представление (IR), оптимизации, виртуальная машина, отладчик и генерация кода.

Для реализации я выбрал C# и .NET. У меня уже был опыт разработки на этой платформе, но в последние годы я в основном писал на Rust. Этот проект стал хорошей возможностью вернуться к C#, немного разнообразить стек и не терять практику с языком.

Именно вокруг этих компонентов и построен csbf — мой компилятор Brainfuck на C#/.NET. Поскольку весь проект компилируется в WebAssembly, он полностью работает прямо в браузере: попробовать.

Как устроен проект

csbf разбит на несколько небольших сборок, и почти все они держатся на одной общей вещи — промежуточном представлении (IR). Парсер переводит исходник в плоский список инструкций Op[], и дальше с текстом уже никто не работает — только с этим списком:

исходник → парсер → IR (Op[]) → оптимизатор → { интерпретатор · транспайлер в Go · JIT }

Ядро Csbf.Core — это парсер, оптимизатор, само IR и интерпретатор (виртуальная машина). Всё остальное подключается к IR как сменные части: кодогенераторы (Csbf.Codegen) реализуют общий интерфейс ICodegen, поэтому новый бэкенд — C или текстовый WebAssembly — это просто ещё одна реализация; отладчик (Csbf.Debugger) и JIT (Csbf.Jit) вынесены в отдельные сборки; а поверх всего лежат два фронтенда — CLI (Csbf.Cli) и плейграунд на Blazor/WASM (Csbf.Web). Показательно, что Csbf.Web подключает всё, кроме Csbf.Jit: в браузере JIT работать негде, поэтому его там просто нет. По сути IR — это контракт: опиши инструкцию один раз, и её сразу понимают оптимизатор, интерпретатор и все кодогенераторы.

Когда синтаксис не главное

Для компилятора современных языков — что основная сложность заключается в парсере, но Brainfuck интересен именно тем, что практически убирает этот этап из уравнения.

Весь синтаксис языка состоит из восьми символов. Поэтому парсер представляет собой обычный проход по исходному коду: каждый символ преобразуется в типизированную инструкцию, а пары [ и ] сразу разрешаются в адреса переходов. После этого остальная часть компилятора уже не работает с текстом и не ищет соответствующую закрывающую скобку — она оперирует единым представлением программы.

Именно с этого момента становятся интересны остальные этапы: промежуточное представление (IR), оптимизации, виртуальная машина, отладчик и генерация кода.

В src/Csbf.Core/Op.cs это выглядит так:

public readonly record struct Op(OpKind Kind, int Arg);

public enum OpKind
{
    AddPtr,   // > <   move the data pointer by Arg
    AddByte,  // + -   add Arg to the current cell
    Out, In,  // . ,   I/O
    Jz, Jnz,  // [ ]   conditional jumps (targets resolved at parse time)
    SetByte,  //       set the current cell (a lowered clear loop)
    ScanPtr,  //       scan by a stride until a zero cell
}

Виртуальная машина, оптимизатор и любой кодогенератор получают этот плоский список операций. Добавь сюда операцию — и она появится везде.

Когда каждый шаг обратим

Виртуальная машина получилась совсем небольшой: указатель инструкций, указатель данных и лента из 30 000 байт с контролем границ. Она умеет выполнять программу по шагам, показывать состояние регистров и памяти и останавливаться на точках останова. Всё это вполне привычно.

Интереснее другое. Благодаря минимализму Brainfuck каждая инструкция изменяет лишь небольшую часть состояния машины: Ip, Dp и, в худшем случае, одну ячейку памяти. Это означает, что любой выполненный шаг можно обратить без повторного исполнения программы или хранения полных снимков состояния. Достаточно сохранить небольшую дельту изменений. Именно так выглядит запись одного шага в src/Csbf.Core/Vm.cs:

// The VM mutates only Ip, Dp, and at most one cell per op, so every step is
// exactly reversible.
private readonly record struct StepDelta(int Ip, int Dp, int CellIndex, byte CellValue);

StepBack() снимает последнюю дельту и восстанавливает изменённые поля. Выполнив N шагов вперёд, а затем N шагов назад, машина оказывается в точности в том же состоянии — с теми же регистрами и содержимым памяти. При этом достаточно хранить лишь кольцевой буфер из этих небольших структур. В CLI это команда back [n], а в браузере — кнопка ⏮. Обратная отладка здесь получается практически бесплатной — как прямое следствие минимализма набора инструкций.

Учим оптимизатор распознавать идиомы

Наивный интерпретатор Brainfuck большую часть времени проводит в циклах, которые выполняют довольно простые действия. Два шаблона встречаются настолько часто, что их имеет смысл считать отдельными инструкциями:

  • [-] и [+] уменьшают (или увеличивают) значение текущей ячейки, пока оно не станет нулём. По сути это операция обнуления ячейки.
  • [>] и [<] двигают указатель, пока не встретится нулевая ячейка. Это аналог линейного поиска (memchr) по памяти.

Оптимизатор csbf распознаёт оба шаблона и заменяет целый цикл одной специализированной инструкцией. Соответствующий код из src/Csbf.Core/Optimizer.cs:

// [-] / [+]: only a ±1 step is a guaranteed clear — e.g. [--] loops
// forever on odd start values, so we deliberately don't lower it.
case OpKind.AddByte when lowerClear && (body.Arg == 1 || body.Arg == -1):
    lowered = new Op(OpKind.SetByte, 0);
    return true;

// [>] / [<] (and collapsed strides like [>>]): scan by a fixed stride.
case OpKind.AddPtr when lowerScan && body.Arg != 0:
    lowered = new Op(OpKind.ScanPtr, body.Arg);
    return true;

Особенно интересен [--] - на первый взгляд этот цикл выглядит как ещё один способ обнулить ячейку. Но это так только для чётных начальных значений. Если значение нечётное, цикл перепрыгивает через ноль, после переполнения становится на 255 и уже никогда не завершается. Именно поэтому оптимизатор сворачивает только циклы с шагом ±1.

Само распознавание занимает буквально несколько строк. Настоящая сложность — убедиться, что преобразование сохраняет семантику программы для всех возможных случаев.

Каждый проход можно включать и выключать независимо. Это удобно не только для измерения производительности, но и для понимания происходящего. Если отключить оптимизацию clear, цикл [-] снова появится в дизассемблере как три отдельные инструкции. Если включить обратно — он снова превратится в один SET 0.

Как это влияет на производительность

Хорошая проверка оптимизатора — классический Brainfuck-мандельброт, выполняющий несколько миллиардов инструкций. Если запустить его сначала без оптимизаций, а затем со всеми проходами, получается:

  • none: 299,72 с
  • all: 60,30 с — ускорение в 4,97×

Количество инструкций после оптимизации уменьшается с 11 451 до 3 619, потому что операции SET и SCAN заменяют целые циклы одной инструкцией. Большая часть выигрыша достигается именно за счёт того, что виртуальной машине приходится выполнять значительно меньше операций. Встроенная команда bench позволяет построить такую таблицу для любой программы.

Тот же IR компилируется в Go

После оптимизации промежуточное представление превращается в плоский список инструкций. Благодаря этому кодогенератор становится почти механическим отображением «одна операция — один case». src/Csbf.Codegen/GoCodegen.cs генерирует обычную Go-программу, и поскольку оптимизатор уже выполнил свою работу, результат больше похож на то что написал человек, а не транслировала виртуальная машина:

// +[-] compiles to:
func main() {
    mem := make([]byte, 30000)
    dp := 0
    mem[dp] += 1
    mem[dp] = 0        // the [-] clear, lowered — not a for-loop
}

Если отключить оптимизации, тот же самый исходный код снова превратится в обычный цикл for mem[dp] != 0 { ... }. Бэкенд при этом остаётся неизменным — меняется только IR.

Тот же IR — сразу в машинный код

Go-бэкенд компилирует IR в другой язык. Последний бэкенд компилирует его сразу в исполнение.

JIT-компилятор генерирует CLR IL внутри DynamicMethod (System.Reflection.Emit) и передаёт его встроенному JIT среды выполнения, который превращает IL в машинный код. Никакого LLVM, внешнего компилятора или дополнительного тулчейна не требуется. Лента передаётся как первый аргумент метода, указатель данных хранится в локальной переменной, а каждая операция транслируется в небольшой набор IL-инструкций, повторяющих логику интерпретатора.

Особенно удобно реализуются циклы. Во время парсинга пары [ и ] уже разрешаются в индексы инструкций, поэтому JIT не ищет парные скобки и не использует стек переходов. Для каждой операции создаётся IL-метка, а переходы сразу указывают на нужную цель:

// One IL label per op index. Jump targets are already op indices (resolved at
// parse time), so a branch points straight at its target — no bracket matching,
// no runtime jump stack.
var labels = new Label[ops.Count + 1];
...
case OpKind.Jz:
    il.Emit(OpCodes.Ldelem_U1);
    il.Emit(OpCodes.Brfalse, labels[op.Arg]);

Проверки границ памяти тоже практически ничего не стоят. Лента представляет собой обычный byte[], поэтому выход указателя за пределы массива автоматически обнаруживается CLR, после чего исключение преобразуется в тот же BrainfuckRuntimeException, который использует интерпретатор.

Именно здесь достигается максимальная производительность. Если полностью оптимизированный интерпретатор выполняет классический Brainfuck-мандельброт почти за полторы минуты, то JIT-версия завершает его примерно за 7,5 секунды — почти в 11,6 раза быстрее. Причём это ускорение получается поверх уже достигнутых ~5× за счёт оптимизаций, потому что JIT работает с тем же оптимизированным IR.

Но есть лишь одно ограничение. Эта часть проекта не работает в браузере. В среде WebAssembly отсутствует JIT на этапе исполнения, поэтому сгенерированный IL просто некому компилировать. По этой причине Jit.IsSupported использует RuntimeFeature.IsDynamicCodeSupported, а при отсутствии динамического кода автоматически выбирается интерпретатор. Полноценный JIT остаётся возможностью десктопной среды.

Запускаем всё это в браузере

Самое приятное, что для всего этого не нужен сервер. Csbf.Core, Csbf.Debugger и Csbf.Codegen компилируются в WebAssembly, а фронтенд на Blazor использует их через IVmIo. Это тот же самый код, что работает в CLI, только теперь он исполняется прямо в браузере.

Во время выполнения можно следить за внутренним состоянием виртуальной машины: ленту с текущим положением указателя, регистры, дизассемблер оптимизированного IR, точки останова, пошаговое выполнение, обратную отладку, переключение проходов оптимизатора, встроенный бенчмарк и генерацию Go-кода.

Brainfuck задумывался как язык-шутка, но его минимализм делает его отличной площадкой для изучения компиляторов. Когда парсер сводится практически к одному проходу по исходному коду, на первый план выходят промежуточное представление (IR), оптимизации, виртуальная машина, кодогенерация и JIT. Оказалось, что восьми инструкций вполне достаточно, чтобы построить вокруг них полноценный компилятор.
