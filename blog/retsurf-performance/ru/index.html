Каждый мегабайт на счету: браузер для консолeй с 1 ГБ ОЗУ

2026-07-01 — Rust · Performance · Servo · Memory — 13 мин чтения

К посту про управление браузер уже загружает современные сайты, а интерфейс полностью управляется с геймпада. На этом можно было бы остановиться — если бы не устройства с 1 ГБ памяти. Этот пост — о том, как сделать так, чтобы оно не только заработало, но и осталось пригодным для использования на очень слабом железе.

Ограничения ниже — те, о которох большинству разработчиков десктопных приложений думать не приходится: общая память (unified memory).

У этих SoC нет выделенной VRAM. GPU хранит framebuffer-ы, тайлы WebRender, декодированные изображения и GL-текстуры в том же пуле памяти (0.5–4 ГБ), из которого CPU выделяет память под JavaScript, DOM и всё остальное. Здесь «уменьшить потребление памяти GPU» и «оставить больше памяти странице» — это не две разные задачи, а одна и та же. Каждый мегабайт, занятый рендерером, отнимается у страницы. На самых маленьких устройствах весь этот бюджет составляет всего 1 ГБ, причём общий ещё и с операционной системой.

Второе ограничение — процессор. Это сравнительно слабые in-order ядра Cortex-A35, A53 и A55 с небольшими L1-кэшами. Они не скрывают промахи кэша за счёт агрессивного спекулятивного выполнения, поэтому лишняя работа здесь означает не просто потерю производительности, а пропущенные кадры и более высокий расход энергии.

Всё, о чём пойдёт речь дальше, основано на полноценного изучении реального кода и исходников Servo, прежде чем считать обоснованной. Большая часть ценности была не в поиске новых оптимизаций, а в понимании того, чего делать не стоит. Полезных изменений оказалось немного, но каждое из них давало постоянный выигрыш.

Этот пост — и про удачные оптимизации, и про те, от которых я сознательно отказался.

Самая дешёвая оптимизация — это аллокации, которые не нужно делать

Уменьшить аллокацию — значит один раз сэкономить память. Удалить её совсем — перестать платить за неё на каждом кадре, каждом ресайзе и при каждом запуске. Поэтому первый проход по коду превратился в поиск вещей, за которые рендерер регулярно делал, хотя пользы от них не было.

Первой такой находкой оказался depth renderbuffer у FBO. WebRender рисует плоские, заранее отсортированные 2D-тайлы, а тест глубины нигде в конвейере не включается. Для complete framebuffer достаточно одного color attachment, поэтому depth-буфер был чистыми накладными расходами: около 2.6 МБ общей памяти при разрешении 1280×720, которые выделялись и переаллоцировались при каждом ресайзе ради механизма, который никогда не использовался.

После удаления путь allocate создаёт только color-текстуру (src/platform/render.rs):

/// (Re)allocate the color texture at `size` and attach it to the FBO. The GL
/// object names are kept stable, so any egui texture registration of the
/// color texture stays valid across resizes. No depth/stencil attachment:
/// WebRender draws flat 2D tiles with the depth test never enabled, so
/// `COLOR_ATTACHMENT0` alone makes the FBO complete.
fn allocate(&self, size: PhysicalSize<u32>) {

Тот же принцип нашёлся и в другом месте - OpenGL возвращает строки изображения снизу вверх, поэтому перед сохранением кадр нужно перевернуть. Старый код делал это, сначала клонируя весь буфер изображения. Новый просто меняет пары строк местами через временный буфер размером в одну строку (stride), избавляясь примерно от 3.5 МБ временных аллокаций при каждом readback-е (src/platform/render.rs):

// Flip vertically: GL returns rows bottom-up. Swap row pairs through a
// single stride-sized scratch buffer instead of cloning the whole frame
// (an odd middle row is already in place, so the half loop skips it).
let stride = w as usize * 4;
let h = h as usize;
let mut tmp = vec![0u8; stride];
for row in 0..h / 2 {
    let top = row * stride;
    let bot = (h - row - 1) * stride;
    tmp.copy_from_slice(&pixels[top..top + stride]);
    pixels.copy_within(bot..bot + stride, top);
    pixels[bot..bot + stride].copy_from_slice(&tmp);
}

Следующая оптимизация влияет уже не при ресайзе, а на каждом кадре. Цветовая текстура, которую egui использует для композитинга браузера, переключилась с LINEAR на NEAREST (src/platform/render.rs):

/// NEAREST is correct because the composite is 1:1 (no scaling, see `ui/mod.rs`),
/// so LINEAR would only burn texture bandwidth without changing the output.
set(gl::TEXTURE_MAG_FILTER, gl::NEAREST);
set(gl::TEXTURE_MIN_FILTER, gl::NEAREST);

Причина проста: egui рисует эту текстуру в масштабе 1:1 — один texel на один пиксель. В таком режиме билинейная фильтрация всё равно читает четыре соседних texel-я, чтобы получить результат, который совпадает с выборкой одного. На памяти с общей шиной это означает десятки мегабайт лишнего трафика каждую секунду без какого-либо выигрыша в качестве изображения. NEAREST выдаёт те же пиксели, просто делает меньше работы.

По отдельности ни одна из этих оптимизаций не выглядит впечатляющей. Но в этом и смысл: они ничего не усложняют, ничего не требуют взамен и продолжают приносить пользу на каждом кадре. Лучшие оптимизации — это те, о которых потом можно просто забыть.

Тюнинг чужого движка

Переписать SpiderMonkey или WebRender это огромная задача. Но у Servo есть десятки настроек, и на ограниченном железе они становятся настоящим инструментом оптимизации — если понимать, что именно ограничивает устройство.

В retsurf они собраны в профили памяти (Embedded, Tight, Balanced, Generous, Android, Desktop), которые выбираются автоматически по объёму RAM. Каждый профиль — это не просто набор отдельных флагов, а согласованный набор компромиссов.

Настройки JIT — самый показательный пример того, как железо само диктует решение. Инстинкт на устройстве с 1 ГБ памяти — отключить JIT: он ведь генерирует машинный код и расходует память. На этих SoC это оказалось ошибкой. Cortex-A35/A53/A55 — in-order ядра, и выполнение JavaScript без baseline JIT оказывается в несколько раз медленнее.

Поэтому профиль Tight, рассчитанный примерно на 1 ГБ RAM, оставляет baseline JIT, но отключает оптимизирующий компилятор (Ion) и его wasm-вариант. Это сокращает расход памяти на сгенерированный код, не обрушивая производительность (src/browser/memory.rs):

// Drop the heavy optimizing JIT; keep baseline JIT. Also drop wasm Ion to
// cut compile-time CPU/memory.
p.js_ion_enabled = false;
p.js_wasm_ion_enabled = false;

Из всех параметров GC только один дал заметный эффект в реальных сценариях — длительность инкрементального слайса. Инкрементальный GC работает короткими шагами с ограничением по времени. Если один такой шаг выходит за бюджет кадра — кадр теряется. Десктопный дефолт в 10 мс вполне приемлем на быстрых процессорах, но уже заметен на Cortex-A35, поэтому в более «тесных» профилях лимит постепенно уменьшается:

p.js_mem_gc_incremental_slice_ms = 4; // ms — keep a GC step well under a frame

Embedded 3, Tight 4, Balanced 6, Generous 8, Android 5; Desktop оставлен на 10. Как и NEAREST, этот параметр ещё просит реального бенчмарка на RK3326/H700 для тонкой настройки — направление точное, точное число — нет.

Цена immediate-mode интерфейса

egui использует immediate-mode UI: весь интерфейс пересобирается каждый кадр. Это делает код проще, но заставляет внимательно относиться ко всему, что выполняется внутри кадра без необходимости.

Хороший пример — домашний экран. Для каждой плитки быстрого доступа он выводит короткую «марку» сайта, а для этого разбирает URL. Повторять этот разбор 60 раз в секунду ради списка закладок, который меняется хорошо если раз в неделю, — именно тот незаметный расход, за который in-order ядра наказывают сильнее всего. Здесь достаточно простого thread-local кэша (src/ui/home.rs):

fn brand_label(url: &str) -> String {
    BRAND_LABELS.with(|cache| {
        if let Some(label) = cache.borrow().get(url) {
            return label.clone();
        }
        let label = compute_brand_label(url);
        cache.borrow_mut().insert(url.to_owned(), label.clone());
        label
    })
}

Но куда интереснее оказались кэши, которые я сознательно не стал добавлять.

Первым кандидатом было кэширование подписей клавиш экранной клавиатуры. На первый взгляд идея выглядела очевидной: десятки клавиш пересчитываются каждый кадр. Но выяснилось, что egui уже кэширует шейпинг текста во внутреннем galley-кэше, а покадрово выполняется только тесселяция — и её всё равно не избежать. Единственное, что можно сэкономить, — несколько небольших строковых аллокаций. Ради этого пришлось бы вводить межкадровый кэш Galley со сложной логикой инвалидирования при изменении масштаба. Сложность оказалась несоизмерима с выигрышем.

Вторым кандидатом был кэш уже готовых Galley для hint-бара. Но здесь проблема ещё серьёзнее: Galley перестаёт быть корректным при изменении pixels_per_point. Масштабирование или поворот пересоздают атлас шрифтов, а вместе с ним меняются UV-координаты текста. Выигрышем была бы повторная раскладка всего нескольких коротких строк, а ценой — целый класс трудноуловимых багов с устаревшими текстовыми кэшами.

Кэш никогда не бывает бесплатным. Он требует правила инвалидирования, а неправильное правило инвалидирования — это уже баг. Поэтому в immediate-mode стоит кэшировать только действительно дорогие операции, вроде разбора URL, а не то, что сам фреймворк уже эффективно оптимизирует.

Запись на диск, не будя устройство

Сначала история записывалась на диск при каждой навигации — целиком history.toml, после каждой загруженной страницы. На устройстве с медленной SD-картой это означало одновременно лишний I/O и лишний износ флеш-памяти.

Самое очевидное решение — перестать писать сразу. Вместо этого запись лишь помечает историю «dirty», а фактическое сохранение откладывается (src/data/history.rs):

/// Record a visit ... Marks the store dirty (the disk write is
/// deferred — see [`Self::flush`]) rather than rewriting the file per visit.
pub fn record(&mut self, url: &str) {
    // ...
    self.dirty = true;
}

Настоящий вопрос здесь не как откладывать запись, а когда её выполнять.

Главный цикл браузера построен вокруг блокирующего wait_event: если пользователю нечего делать, процесс действительно засыпает, а не просыпается десятки раз в секунду проверять таймеры. Именно это во многом и обеспечивает нормальное время работы от батареи.

Поэтому стандартное решение — «периодически сбрасывать историю по idle-таймеру» — здесь оказывается неправильным. Такой таймер неизбежно будил бы цикл по расписанию, разрушая оптимизацию, на которой держится модель энергопотребления.

Вместо этого сохранение должно происходить только тогда, когда приложение и так уже проснулось. История сбрасывается с троттлингом раз в пять секунд, но исключительно во время кадров, которые цикл всё равно обрабатывает. Дополнительно выполняется гарантированный сброс при закрытии меню и завершении работы (src/app/mod.rs):

// Recording only marks history dirty; flush it on a throttle so a busy
// browsing burst collapses to one write per interval. This piggybacks
// on frames the loop is already awake for — it never schedules an idle
// wake (the blocking wait stays battery-efficient). A clean exit and
// menu close flush the remainder.
if self.last_history_flush.elapsed() >= HISTORY_FLUSH_INTERVAL {
    self.ui.flush_history();
    self.last_history_flush = Instant::now();
}

Компромисс здесь простой: при жёстком завершении процесса можно потерять до пяти секунд истории посещений. Для истории это вполне приемлемо.

Зато не пришлось трогать сам цикл ожидания. Это хороший пример того, что существующая архитектура тоже может быть оптимизацией. Если именно блокирующий wait_event позволяет устройству прожить на одном заряде заметно дольше, то любая новая оптимизация должна строиться вокруг него, а не бороться с ним.

Оптимизация, которую нельзя использовать везде

panic = "abort" — почти бесплатная оптимизация для релизной сборки. Она уменьшает бинарник, убирает таблицы раскрутки стека, сокращает объём памяти после mmap и немного ускоряет холодный старт. Для консольной версии retsurf это именно то, что нужно.

Но только для неё.

У проекта есть ещё одна ARM-сборка — универсальная, с включённым WebGL, рассчитанная на запуск вне PortMaster. Именно она не может использовать panic = "abort".

Причина уходит к предыдущему посту. Для совместимости surfman с EGL 1.4 использует catch_unwind, чтобы превратить внутреннюю панику библиотеки в обычное отсутствие поддержки WebGL. При panic = "abort" это становится невозможным: стек больше не раскручивается, паника не перехватывается, а процесс сразу завершается.

Поэтому panic = "abort" включается только в консольной CI-сборке, где WebGL полностью исключён на этапе компиляции. Общий профиль проекта намеренно оставляет эту настройку выключенной (Cargo.toml):

# `panic = "abort"` is deliberately NOT here: it makes `catch_unwind` inert and
# would break the surfman EGL-1.4 probe (`platform/render.rs`) on webgl builds.
# The handheld CI build (webgl off) sets it via env var, where it's safe.

Оно безопасно только на единственной сборке, где catch вырезается компиляцией целиком — консольной сборке с выключенным webgl, — и там его включает переменная окружения в CI, а не общий профиль. А что попало в каждую сборку:

strip = true
debug-assertions = false
overflow-checks = false

Самая полезная из них — strip. CI автоматически удаляет символы только из Linux-бинарника, но локальные сборки и Android-библиотеки такого шага не проходят. Без этой настройки .so уезжал с полной таблицей символов, занимая больше памяти и медленнее загружаясь при холодном старте.

Этот пример хорошо показывает, что оптимизации редко бывают универсальными. panic = "abort" действительно уменьшает бинарник — но только там, где его можно позволить себе без потери корректности.

Один бинарник не запускается на всех ARM

В начале поста я объединил A35, A53 и A55 в одну категорию — слабые in-order ядра. Для разговора о памяти это вполне справедливо.

Для генерации машинного кода — уже нет.

A35 и A53 реализуют ARMv8.0-A. A55 — ARMv8.2-A: появляются LSE-атомики, FP16, dot product и другие инструкции, которых старые ядра просто не понимают. Если собрать Servo с -C target-cpu=cortex-a55, LLVM начнёт свободно использовать эти инструкции по всему бинарнику — в том числе глубоко внутри многопоточного кода.

Такой бинарник не станет медленнее на A35. Он вообще не запустится.

Первая же инструкция, которой процессор не знает, закончится SIGILL, причём ещё до того, как управление дойдёт до main(). Здесь target-cpu — уже не совет компилятору, а выбор минимальной ISA, под которую собирается программа.

Из этого довольно быстро следует единственное безопасное решение: одного бинарника не достаточно.

CI собирает консольную версию три раза — отдельно для каждого семейства процессоров:

matrix:
  include:
    # ARMv8.0-A, in-order. RK3326. Runs on A53 too (same ISA).
    - cpu: cortex-a35
    # ARMv8.0-A, in-order, crypto OFF. Safe v8.0 baseline + default fallback.
    - cpu: cortex-a53
    # ARMv8.2-A: LSE atomics, fp16, dotprod + crypto. SIGILLs on the v8.0
    # cores above — must stay a separate binary.
    - cpu: cortex-a55

Сам флаг — одна строка, применяемая на каждую матричную задачу; остальное в этой сборке — тот же CI-only профиль производительности, что описан в прошлом разделе (fat LTO, один codegen-юнит, panic = "abort", потому что webgl выключен):

RUSTFLAGS: "-C target-cpu=${{ matrix.cpu }}"

Три бинарника решают сборку. И создают новую проблему: теперь устройству надо выбирать. А выбрать изнутри процесса оно не может — к моменту, когда доходит до main, A35, которому подсунули A55-бинарник, уже упал где-то в стартовом пути. Выбор должен произойти слоем выше, в лаунчере, до того как не тот бинарник вообще будет запущен.

Порты PortMaster запускаются через shell, так что диспетчеризация живёт в Retsurf.sh. Он читает id «CPU part» ARM из /proc/cpuinfo и exec-ает подходящую сборку (portmaster/Retsurf.sh):

CPU_PART="$(grep -m1 -i 'CPU part' /proc/cpuinfo | grep -oiE '0x[0-9a-f]+' | head -1 | tr 'A-Z' 'a-z')"
select_binary() {
  case "$CPU_PART" in
    0xd05) grep -qw atomics /proc/cpuinfo && echo "retsurf.a55" || echo "retsurf.a53" ;;
    0xd04) echo "retsurf.a35" ;;
    *)     echo "retsurf.a53" ;;   # A53 and any unrecognized CPU
  esac
}

В этом коде есть две важные детали.

Первая — путь по умолчанию. Любой неизвестный процессор получает сборку a53. Это самая консервативная цель: ARMv8.0 без дополнительных расширений. Ошибка в эту сторону означает лишь недополученную производительность. Ошибка в сторону A55 означает мгновенный SIGILL.

Вторая — дополнительная проверка перед запуском A55-сборки. Одного CPU part недостаточно: лаунчер ещё требует наличия флага atomics в /proc/cpuinfo. Только после этого он считает, что процессор действительно поддерживает инструкции ARMv8.2, ради которых вообще существует отдельная сборка.

Есть и четвёртый вариант — универсальная ARM-сборка вне PortMaster. Она вообще не использует target-cpu, оставляет включённым WebGL и работает на любом ARMv8.0+. Это самый медленный вариант, зато единственный, который можно распространять отдельно от конкретного устройства.

Оптимизация, которая редко приносит пользу

Не каждая найденная оптимизация действительно что-то меняет на консоли.

Во время поиска я заметил, что параметры сэмплирования и wrap у color-текстуры переустанавливаются при каждом ресайзе FBO, хотя glTexImage2D их не сбрасывает, так что переустановка была просто повторением одной и той же работы.

Теперь они выставляются один раз при создании текстуры (src/platform/render.rs):

Выводы

  1. На системах с общей памятью производительность и память — это один ресурс. Здесь нет отдельной VRAM, в которой можно «спрятать» лишние аллокации. Каждый мегабайт, который удерживает GPU, — это мегабайт, который уже не сможет использовать JavaScript или DOM. Оптимизация графики и оптимизация памяти приложения оказываются одной и той же задачей.
  2. Самая дешёвая оптимизация — удалить работу целиком. Не уменьшить глубинный буфер, а убрать его. Не сделать копию кадра дешевле, а перестать её делать. Сокращение накладных расходов почти всегда менее выгодно, чем их полное исчезновение.
  3. Оптимизировать нужно систему целиком, а не отдельный участок кода. Idle-таймер уменьшил бы число операций записи, но одновременно ухудшил бы модель энергосбережения, построенную вокруг блокирующего.
  4. Корректность задаёт предел производительности. panic = "abort" действительно уменьшает бинарник, но не может использоваться там, где нужен catch_unwind. Иногда привлекательная оптимизация оказывается несовместимой с архитектурой проекта.
  5. Один target не всегда означает одну платформу. ARMv8 — это семейство, а не конкретный процессор. Для A35, A53 и A55 разница в ISA уже настолько существенна, что безопаснее распространять несколько бинарников, чем пытаться подобрать один «универсальный».

Большинство изменений из этой статьи небольшие по отдельности. Где-то исчезли несколько мегабайт памяти, где-то — несколько GL-вызовов или лишняя аллокация, а где-то лучшим решением оказалось вообще ничего не менять. Но именно так обычно и выглядит оптимизация на реальном проекте: не одна большая идея, а десятки маленьких решений, каждое из которых убирает немного лишней работы. Именно из них в итоге и складывается разница между программой, которая просто запускается, и программой, которой действительно приятно пользоваться.
